## 06-06 调查：Token 消耗根因

**触发**：老板质疑"是不是我频繁让你汇报→读整个 context"

**结论**：

| 假设 | 实际 | 占比 |
|---|---|---|
| 5min cron 反复读 context | ❌ 不成立——6/5 全天 7 个 cron 总共 ~150k input | < 5% |
| **主因：dashboard session 累积** | ✅ **早上 5.5h session 累计 1340 万 input，单轮峰值 148 万** | ~80% |
| spawn 长 prompt | 玛门/路西法 1 task 1 session 已控住 | ~10% |
| 大文件 read | 偶发，自查违规 1 次 | < 5% |

**根因**（来自 OpenClaw 文档研究）：
- M3 1M context 永远不会自动触发 compaction（auto-compaction 只在「接近上限」或「overflow 错」时触发）
- 长 dashboard session 跨小时累积 history
- 跨 session 交接时 context 继承（早上 session 结束时 **接班 session 起步就 19 万 input**）
- 早 6/5 session (3d4960a1) 27 轮 input 分布：200k → 1.48M → 498k（每轮都重喂）

**OpenClaw 已有但没启用的工具**（来自 `docs/concepts/compaction.md` + `session-pruning.md`）：
1. **`/compact`** — 手动压缩，给定主题保留
2. **`maxActiveTranscriptBytes`** + `truncateAfterCompaction: true` — 字节级自动 compaction
3. **`contextPruning.mode: "cache-ttl"`** — 修剪老 tool result（轻量）
4. **`toolResultMaxChars`** — 默认 16k-64k（已生效）
5. **`/usage full`** — 每条回复显示 cost

**6/6 改进**（已落 AGENTS.md 5c 子规则 7、8）：
- 子规则 7：**撒旦主动 `/compact`**，1.5-2h / 30 轮 / 大工具爆发后
- 子规则 8：**3h 重开 session** — 老板自己控；撒旦不主动 reset，但开工前 `/status` 看 context，>50% 提醒

**待老板拍板的 config 改动**（撒旦不动手）：
- `openclaw.json` 加 `compaction.maxActiveTranscriptBytes: 5242880`（5MB 触发）+ `truncateAfterCompaction: true`
- `openclaw.json` 加 `contextPruning: { mode: "cache-ttl", ttl: "5m" }`
- 改完需 `openclaw gateway restart`

## 教训沉淀
1. **5min cron 不是元凶**——是 dashboard session 长度元凶
2. **M3 1M context 是陷阱**——不会自动 compact，反而鼓励 session 越长越省钱（错！）
3. **/compact 必须主动**，不能等 auto
4. **OpenClaw 自带工具够用**——`/compact` + `pruning` + `maxActiveTranscriptBytes` 组合拳
