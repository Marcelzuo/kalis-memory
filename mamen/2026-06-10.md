## 我（撒旦）犯的错：_redirects 死循环

**错规则**：
```
/* https://kalistorik.com/:splat 301
```

**错因**：
- CF Pages `_redirects` 语法是 `<source> <target> [status]`
- **不支持按域条件**（per-domain redirect）
- `/*` 匹配所有请求路径 → 所有请求 301 跳 `kalistorik.com/<原路径>` → apex 自身也被规则命中 → **死循环**

**代价**：站点死了 5min+

**修复**：删 _redirects 第 5 行，恢复 4 行原状（仅 4 条 social-*.html → /404.html）

**教训**（铁律 2）：**CF Pages `_redirects` 不支持 per-domain redirect**。任何域级跳转（www→apex、HTTP→HTTPS 跨域）必须用：
- **CF Dashboard → Rules → Page Rules**（推荐，UI 友好）
- 或 **Bulk Redirects**（批量）
- 或 **CF Worker**（复杂场景）

**教训**（铁律 3）：**改配置类文件必须派 1 验 1 完整链路**。这次 deploy 完只 curl 了 index，没 curl 其他路径，死循环没立刻暴露。下次改 `_redirects` / `nginx.conf` / `.htaccess` 等配置必须 1min 等边缘传播 + 至少 3 个不同路径 curl 验（首页 / 已知旧路径 / social-* 等）。

## CF Pages wrangler deploy 关键参数

**核心坑**：CF Pages 项目 `kalistorik` 的 `production_branch` = `main`，但本地 git 当前在 `master` 分支。

**wrangler deploy 行为**：
- 不带 `--branch` 参数 → 读当前 git 分支（master）→ 推到 master 环境
- 因为 production_branch = main，所以 master = **preview**（URL 形如 `<hash>.kalistorik.pages.dev`）
- 推到 production 必须**显式指定 `--branch=main`**

**deploy 完整命令**：
```bash
cd /Users/zuo/.openclaw/workspace/kalis/dist
wrangler pages deploy . --project-name=kalistorik --branch=main
```

**教训**（铁律 4）：**deploy 后必须 `wrangler pages deployment list --project-name=<proj>` 验 Production/Preview 标记**，不是只看 deploy 输出说"成功"。

## 验收模板（更新版）

部署完成 / 修复完成 / 任何 HTTP 改动后：

```bash
# 1. DNS 验证
dig +short CNAME <域>
dig +short A <域>  # 应该是 CF IP

# 2. 生产域 MD5 验版本
curl -sS https://<域>/ | md5

# 3. 关键路径 HTTP 码
curl -sS -I https://<域>/<路径>  # 期望的 200/301/404

# 4. 关键内容 grep
curl -sS https://<域>/ | grep -c "<关键文本>"

# 5. 跨域跨地区边缘
for cc in US FR DE JP; do
  curl -sS -H "CF-IPCountry: $cc" https://<域>/ | md5
done
```

**新增**（vs 之前 17:40 模板）：
- DNS 验证必须放在最前
- 跨地区 MD5 一致（CDN 边缘）
- 多路径 curl（不能只看首页）
